# PARA File Organizer Skill 生成用 brief

このプロンプトを `/writing-skills`（superpowers）または同等の skill 生成エージェントに渡してください。あなた専用の汎用 PARA file-organizer skill が生成されます。

---

## スキルメタ情報

- **スキル名**: `para
- **1 行 description**: Obsidian vault のファイルを PARA メソッド（Projects / Areas / Resources / Archive）に基づいて自動分類し、wikilink を壊さずに移動する skill
- **動作環境**: Claude Code / Claudian / Cowork / Claude.ai Pro 以上（Skills 機能が必要）

## 目的

任意の Obsidian vault のフォルダ構造を、PARA メソッドに基づいて整理する。Vault の具体的なパスやテンプレート構造に依存せず、ユーザーの環境に汎用的に適用できる skill を生成する。

## PARA メソッドの定義（参照源: Tiago Forte 公開メソッド）

PARA は Tiago Forte（@fortelabs / fortelabs.com）が提唱したフォルダ整理メソッドで、すべての情報を以下の 4 カテゴリに分類する:

- **Projects（P）**: 終わりがある活動。明確なゴールと期限があるもの。例: ポートフォリオサイト構築、資格試験準備
- **Areas（A）**: 終わりがない継続的責任領域。維持し続ける基準・状態がある。例: 健康管理、家計、家族
- **Resources（R）**: 関心トピック。将来役立つかもしれない参考資料の集積。例: 興味ある技術、読みたい記事
- **Archive（K）**: 完了したもの。Project の終了物 / 関心が消えた Resource

### 評価順序（actionability 順）

ファイルを分類する際は、以下の順番で判定する:

1. これは今アクション可能な Project か？ → Yes なら `200_Projects/` に分類
2. 違うなら、これは継続維持する Area か？ → Yes なら `300_Areas/` に分類
3. 違うなら、参考にする Resource か？ → Yes なら `400_Resources/` に分類
4. 違うなら、終わったものとして Archive へ → `900_Archive/` に分類

迷ったらより actionable な側（より上）に分類する。

### フォルダ命名規則

- 番号プレフィックス + 英語名: 000_me`,`100_inbox`, `200_Projects`, `300_Areas`, `400_Resources`, `900_Archive`
- 番号プレフィックスは Obsidian サイドバー上の並び順を固定するためのもの
- Archive は 900 にして他カテゴリと離す（「使っているフォルダ」と「終わったフォルダ」の視覚的分離）
- ファイル名・サブフォルダ名は英語推奨（AI 操作の安定性のため）
- サブフォルダは「1 つのまとまり = 1 instance」（具体的な 1 案件 / 1 領域 / 1 トピック）に対応させる
- `Stuff` `Misc` `Things` 等の曖昧名は禁止

## スキルの動作要件

### 1. Dry-run モード（必須・デフォルト）

- 実行時、まず「何をどこに移動するか」の一覧を出力する
- 各ファイルの分類判断根拠を 1 行で記載する
- ユーザー承認を待ってから実行する。承認前にファイルを動かさない

### 2. Ask vs Decide の境界（明示）

**質問する場面（必ずユーザーに確認）:**
- ファイルが Project と Area の間で曖昧（特に財務・法務・税務・期限関連）
- スクリーンショットの目的がファイル名・内容から推測できない
- ファイルが読めない（暗号化、破損、対応外フォーマット）
- 複数のアクティブな Project に属しうる

**自律判断する場面（質問せず分類）:**
- ファイル名と内容から 1 カテゴリに明確にマッピングされる
- 明らかに ephemeral（meme、通知スクリーンショット、一時ダウンロード等）

### 3. wikilink を壊さない移動

- フォルダ間移動のみの場合、Obsidian の wikilink はファイル名ベースで解決されるため、特別処理不要で維持される
- ファイル名変更が必要な場合は、ユーザーの vault 内に rename スクリプト（例: `rename-note.sh` のようなもの）があるか先に確認し、なければユーザーに確認する。Vault 全体の wikilink 追従が必要な場合は事前にユーザーへ通知する
- 新規リンク作成時は `[[wikilink]]` 構文を使う。`[text](path)` 形式は使わない

### 4. frontmatter 補完（任意）

- frontmatter がないファイルには最低限の frontmatter を追加する: `title`, `created`, `tags`（空配列可）
- 既存 frontmatter があるファイルは変更しない

### 5. サブフォルダ追加・構成レビューの提案

ファイル分類だけでなく、対象 vault の現状フォルダ構成も分析し、以下を提案する。実行は必ずユーザー承認後（Dry-run の一部として提案を出す）:

**サブフォルダ追加の提案:**
- INBOX または各 PARA カテゴリ直下に同じテーマのファイルが **5 個以上**溜まっている場合、そのテーマ名でサブフォルダ作成を提案する
- サブフォルダ名は「1 instance = 1 具体名」のルールで命名する（例: `Q1-Launch` / `Health` / `Web-Dev-Resources`）
- 提案後、ユーザー承認を得てからサブフォルダ作成 + ファイル移動を実行する

**アンチパターン検出と改善提案:**
- **空の中間フォルダ**（中身がない箱、例: `200_Projects/2026/` を作ったが中が空）→ 削除提案
- **1 つしかないカテゴリ**（例: `200_Projects/Work/Q1-Launch/` で `Work/` の中身が Q1-Launch だけ）→ 階層を 1 つ減らす提案
- **曖昧名**（`Stuff` / `Misc` / `Things` 等、名前で意味が読み取れないフォルダ）→ 具体名へのリネーム提案
- **先回り作成された空フォルダ**（まだ存在しない案件のために先に作った空箱）→ 削除または「実際に必要になってから作る」と促す
- **意味のない深い階層**（各階層に意味がない 3-4 階層）→ 階層削減提案

### 6. 完了後サマリー

実行後に以下を報告する:
- 分類結果（カテゴリ別の件数）
- サブフォルダ追加・構成変更で実行した内容
- 判断が微妙だったファイルのリスト + 判断根拠
- 自律判断ではなく質問して確定したファイルのリスト
- 未確定で保留したファイル・構成提案のリスト

## スコープ

### in（このスキルが扱う）

- ユーザーが指定した任意フォルダ配下のファイル分類
- PARA カテゴリへの分類提案
- actionability 評価
- Ask vs Decide の境界判断
- **サブフォルダ追加の提案（5+ ファイル溜まったテーマで instance ベース命名）**
- **既存フォルダ構成のアンチパターン検出と改善提案（空中間フォルダ / 1 件カテゴリ / 曖昧名 / 先回り空フォルダ / 意味なし深階層）**
- Dry-run 出力 → 承認 → 実行（ファイル移動・サブフォルダ作成・構成変更すべて含む）
- 完了後サマリー

### out（このスキルは扱わない）

- 特定 vault 固有のパス・命名規則の自動学習（汎用 skill のため）
- ナレッジグラフ層（atomic notes, insights/, ops/scripts/ 等の独自ディレクトリ）への書き込み
- 特定の Obsidian プラグイン（Bases, Canvas, obsidian-skills 等）への依存
- ユーザー承認なしの構造変更（rename / 統合 / 分割は必ず承認後）

## 検証シナリオ

スキル生成後、以下のシナリオで動作確認する:

1. **基本分類**: 任意フォルダにテストファイル 5 個（明らかな Project / Area / Resource / Archive 各 1 個 + 曖昧 1 個）を置き、dry-run で正しく分類提案されるか
2. **Ask vs Decide**: 期限付き税務書類 / 通知スクリーンショット / 暗号化ファイル を置き、それぞれ「質問」「自律判断（Archive）」「質問」と動作するか
3. **wikilink**: wikilink を含むノートを別フォルダに移動し、リンクが壊れていないか確認
4. **frontmatter**: frontmatter なしのノートに正しく追加されるか
5. **承認フロー**: dry-run 出力後、ユーザーが「No」と返したらファイルが移動されないか
6. **サブフォルダ提案**: inbox に同テーマのファイル 6 個を置き、dry-run でサブフォルダ追加提案が出るか
7. **アンチパターン検出**: 空の中間フォルダ / `Stuff` 名フォルダ / 1 件しかないカテゴリ を含む構造に対し、それぞれの改善提案が出るか

## non-goals（このスキルは以下を行わない）

- ファイル内容の AI 要約・タグ自動生成（別スキルの仕事）
- vault 内の重複ファイル検出
- バックアップ作成（ユーザー側で git 等を使う前提）
- 特定 OS 依存処理（macOS 限定 / Windows 限定 等）
- vault 構造そのものの再設計（PARA カテゴリ内での分類に留める）

このスペックに基づいて、上記要件をすべて満たす skill を生成してください。生成完了後、検証シナリオ 1-7 をどこで確認できるか、テストファイルの配置例とともに教えてください。
