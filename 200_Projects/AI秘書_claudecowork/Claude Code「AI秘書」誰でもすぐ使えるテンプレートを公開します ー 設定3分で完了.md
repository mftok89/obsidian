## はじめに

Claude Code（Anthropicが提供するCLIベースのAIアシスタント）を使って、自分専用の「AI秘書」を作りました。

- 毎朝、スケジュール・メール・タスクをまとめたブリーフィングを出してくれる
- メールを優先度で分類し、返信案まで作ってくれる
- 会議前に準備メモを作り、会議後に議事録を整理してくれる
- 調べ物を頼むと、ソース付きのリサーチレポートにまとめてくれる
- 仕事を依頼すると、自動でプロジェクトとして登録・管理してくれる

これらの機能を、CLAUDE.md とフォルダ構成だけで実現しています。コードは1行も書いていません。

GitHubの公開リポジトリとして公開したので、Claude Codeを使える環境があればclone後すぐに試せます。

- AI秘書テンプレートの全体設計と、なぜこの構成になったのか
- 6つの機能（ブリーフィング、メール整理、会議準備、リサーチ、プロジェクト管理、自動学習）の具体的な使い方

## 対象読者

- Claude Code を使ったことがある、またはこれから使いたい方
- 日々の業務（メール、スケジュール、タスク管理）を効率化したい方
- AIアシスタントの「指示の設計」に興味がある方

## 前提条件

- **Claude Code** がインストール済みであること（ [公式ガイド](https://docs.anthropic.com/en/docs/claude-code/overview) ）
- **Anthropicの有料プラン** （Claude Pro / Max）が必要です。無料プランではClaude Codeは利用できません
- **macOS / Linux**: そのまま利用可能です
- **Windows**: WSL（Windows Subsystem for Linux）環境が必要です
- メール・カレンダー連携を使う場合は、別途MCPサーバーの設定が必要です（なくても動きます）

## なぜ作ったのか

Claude Codeは、CLAUDE.mdというファイルにルールを書いておくと、そのルールに従って動いてくれます。これはコードを書くためだけの機能ではなく、「AIの振る舞いを設計できるプラットフォーム」でもあります。

ただ、いざ「AI秘書を作ろう」と思うと、いくつかの壁にぶつかります。

1. **CLAUDE.mdに何を書けばいいか分からない** — 「秘書として動いて」だけでは精度が出ません
2. **ファイルが散らかる** — AIが作った議事録やレポートがフォルダ直下に溜まっていきます
3. **毎回同じ説明をする** — 「メールを優先度で分けて」と言っても、基準が毎回ブレます
4. **使うほど賢くなってほしい** — 好みを覚えてくれないと、毎回やり直しが発生します

これらを解決するために、CLAUDE.mdの書き方、フォルダ構造、業務プロトコル、学習の仕組みをセットで設計しました。

## 全体設計 — 3層アーキテクチャ

AI秘書の設計は、3つの層に分かれています。

<iframe src="https://qiita.com/embed-contents/mermaid#qiita-embed-content__dc8256fba896a3a8d7effd1c8c970bb1" frameborder="0" height="478"></iframe>

この3層に分けた理由は明確です。

CLAUDE.mdに全部書くと500行を超えて、AIが指示を見落とす確率が上がります。逆に、CLAUDE.mdを最小限にして詳細をprotocols/に分離すれば、CLAUDE.mdは「どのプロトコルを使うか」の判断だけに集中できます。

実際のCLAUDE.mdは65行しかありません。

### CLAUDE.mdの中身

```markdown
# AI Secretary

あなたはユーザーの秘書・アシスタントである。
ユーザーの時間を最大化し、判断の負荷を最小化することが使命である。

## 初回セットアップ

\`config/profile.md\` の「名前」が（未記入）の場合、\`protocols/onboarding.md\` に従って
対話形式でプロフィールとツール設定を完了させる。他の業務より優先する。

## 行動原則

- 結論ファースト。前置き・挨拶・絵文字は不要
- 確認より実行を優先する。「こうします」と宣言してから動く
- 確認が必要な場面: 金銭・契約・セキュリティ、または判断が分かれる重要事項のみ
- 推測で書かない。不明な点は明示するか質問する

## タスクルーティング

| トリガー | プロトコル | 説明 |
|----------|-----------|------|
| 「おはよう」「今日の予定」 | protocols/daily-briefing.md | 今日のスケジュール・メール・タスクを要約 |
| 「メール確認」 | protocols/email-triage.md | メールを優先度分類し、対応案を提示 |
| 「会議準備」「議事録」 | protocols/meeting-prep.md | 会議前の準備 or 議事録作成 |
| 「調べて」「リサーチ」 | protocols/research.md | 構造化されたリサーチ |
| 新規タスク・案件の依頼 | protocols/project-mgmt.md | プロジェクトとして登録・管理 |
```

ポイントは「タスクルーティング」のテーブルです。ユーザーの発言に含まれるトリガーワードに応じて、どのプロトコルファイルを参照するかを明示的にマッピングしています。

これにより、「おはよう」と言えば朝ブリーフィングが走り、「メール確認して」と言えばメールトリアージが走ります。AIの判断のばらつきを減らし、プロトコルに従いやすくするための仕組みです。

## フォルダ構成 — PARAメソッド

ファイル管理にはPARAメソッドを採用しています。

PARAとは、Tiago Forte氏が考案した情報整理のフレームワークで、すべての情報を4つのカテゴリに分けるルールです。

| カテゴリ | 意味 | AI秘書での対応フォルダ |
| --- | --- | --- |
| **P** rojects | 期限とゴールがある仕事 | `projects/` |
| **A** reas | 終わりがなく継続的に管理するもの | `areas/` |
| **R** esources | いつか役立つ参考情報 | `resources/` |
| **A** rchive | 完了した・使わなくなったもの | `archive/` |

たとえば、「セミナー用のプレゼンを4/15までに作る」は期限とゴールがあるので **Projects** です。「毎朝のブリーフィング記録」は終わりがないので **Areas** です。「業界のリサーチ結果」は参考情報なので **Resources** です。

AI秘書がファイルを作成するたびに、この4分類に従って自動的に正しいフォルダに保存します。ユーザーが手動でファイルを移動する必要はありません。

```text
ai-secretary/
├── CLAUDE.md              ← AI秘書の行動原則（65行）
├── config/                ← ユーザー固有の設定
│   ├── profile.md           名前・役職・仕事スタイル
│   └── tools.md             ツールプリセット（有効/無効切替）
├── protocols/             ← 業務プロトコル（6ファイル）
│   ├── onboarding.md        初回セットアップ
│   ├── daily-briefing.md    朝ブリーフィング
│   ├── email-triage.md      メールトリアージ
│   ├── meeting-prep.md      会議準備・議事録
│   ├── research.md          リサーチ
│   └── project-mgmt.md      プロジェクト管理
├── memory/                ← AI学習記録（自動蓄積）
│   ├── learnings.md         セッション毎の気づき
│   └── patterns.md          ユーザーの好み・頻出パターン
├── projects/              ← PARA: P（進行中の案件）
├── areas/                 ← PARA: A（日々の記録）
├── resources/             ← PARA: R（参考情報）
└── archive/               ← PARA: A（完了した案件）
```

## セットアップ — cloneして話しかけるだけ

### 1\. リポジトリをクローン

```bash
git clone https://github.com/nogataka/ai-secretary.git
cd ai-secretary
```

### 2\. Claude Codeを起動

```bash
claude
```

### 3\. 自動で対話型セットアップが始まる

`config/profile.md` が未記入の状態でClaude Codeを起動すると、AI秘書が自動でセットアップを開始します。

```text
AI秘書: AI秘書のセットアップを始めます。いくつか質問させてください。

AI秘書: お名前（呼び方）を教えてください
あなた: 田中です

AI秘書: 役職・お立場を教えてください
あなた: マーケティング部長です

AI秘書: 主な責務を2〜3つ教えてください
あなた: マーケ戦略の立案、チームマネジメント、予算管理

AI秘書: 報告の好みを教えてください: 簡潔 / 詳細 / 要点+根拠
あなた: 簡潔で

AI秘書: 普段使っているツールを教えてください（複数可）
あなた: GmailとGoogleカレンダーとSlack

AI秘書: セットアップが完了しました。何から始めますか？
```

このやりとりの結果が `config/profile.md` と `config/tools.md` に保存されます。所要時間は3〜5分です。

このオンボーディングの設計にもこだわりがあります。「profile.mdを手動で編集してください」とREADMEに書くだけでは、非エンジニアはそこで離脱します。「対話で埋まる + 手動でも編集できる」の両対応にしたのは、技術レベルを問わず使えるようにするためです。

## 機能ガイド — 6つの機能を詳しく解説

### 機能1: 朝ブリーフィング

「おはよう」と話しかけるだけで、今日1日の全体像を30秒で把握できるレポートを出してくれます。

```text
あなた: おはよう
```

AI秘書は以下の手順で情報を収集し、レポートを作成します。

<iframe src="https://qiita.com/embed-contents/mermaid#qiita-embed-content__8b7a87bab99517b3c9214534049cda64" frameborder="0"></iframe>

出力例はこのような形式です。

```markdown
## 今日のポイント
- 15時のA社ミーティングで予算承認の判断が必要
- B社からの見積もり依頼に本日中に返信が必要

## スケジュール
- 10:00 チーム朝会（準備: 先週の数値レポートを確認）
- 13:00 ランチミーティング 佐藤さん
- 15:00 A社 定例ミーティング（準備: 予算案の最終確認）

## メール（要対応）
- 「見積もりの件」from B社 山田様 — 見積もり依頼。本日中に返信推奨

## タスク・締切
- PRJ-003 提案書作成: 納期 4/5（残り6日）

## 判断待ち
- B社見積もり: 標準プランで回答してよいか？
```

ここで「判断待ち」セクションが地味に便利です。AI秘書が処理できなかった判断事項を明示的にリストアップするので、自分が考えるべきことだけに集中できます。

ブリーフィングは `areas/daily-briefings/2026-03-30.md` のように日付付きで自動保存されるため、後から振り返ることもできます。

カレンダーやメールのMCP連携がなくても使えます。AI秘書が「予定を貼り付けてもらえますか？」と聞くので、カレンダーアプリからコピペするだけでOKです。

### 機能2: メールトリアージ

未読メールを4段階の優先度で自動分類し、高優先度のメールには返信案まで作成します。

```text
あなた: メール確認して
```

MCP連携がない場合は、メール画面のスクリーンショットやテキストを貼り付けて「これを整理して」と伝えれば同じように動きます。

分類基準はprotocols/email-triage.mdに定義されています。

| 優先度 | 条件 | AIの対応 |
| --- | --- | --- |
| 高 | 返信期限あり、金銭関連、クレーム | 要約 + 返信案を作成 |
| 中 | 返信は必要だが急ぎでない | 要約 + 推奨アクション |
| 低 | CC、ニュースレター | 1行要約のみ |
| 無視 | スパム、不要な自動通知 | 件数だけ報告 |

出力例です。

```markdown
## メールトリアージ（2026-03-30）

### 高（即対応: 1件）
1. 「見積もりの件」from B社 山田様（9:15）
   - 要点: 先日の打ち合わせの件で見積もり依頼。今週中の回答を希望
   - 推奨: 標準プランの見積もりを本日中に送付
   - 返信案:
     山田様
     お世話になっております。
     見積もりの件、承知いたしました。本日中にお送りいたします。

### 中（当日対応: 1件）
1. 「【週報】マーケチーム」from 鈴木
   - 要点: 今週のKPI達成率85%。SNS施策が好調
   - 推奨: 確認済みの返信。SNS施策の継続を承認

### 低（確認のみ: 1件）
- 「セミナーのご案内」 — AI活用セミナー（4/15開催）

### 無視（0件除外）
```

返信案はそのまま使うこともできますし、「もう少しカジュアルにして」「締切を強調して」などの修正指示も可能です。この分類基準はprotocols/email-triage.mdを編集すれば自由に変更できます。たとえば「特定のクライアントからのメールは常に高優先」といったカスタマイズが可能です。

### 機能3: 会議準備・議事録

会議の前後を2つのモードでサポートします。

#### 会議前: 準備メモ

```text
あなた: 15時のA社ミーティングの準備して
```

AI秘書がカレンダーから会議情報を取得し、過去の議事録から持ち越し事項を抽出して、準備メモを作成します。

```markdown
## A社 定例ミーティング（2026-03-30 15:00）

### 参加者
- 田中（自社・マーケ部長）
- A社 佐々木様（営業部長）

### 議題
- Q1実績の共有
- Q2のキャンペーン予算の承認

### 前回からの持ち越し
- SNSキャンペーンの効果測定結果 → 田中が共有予定

### 準備事項
- Q1実績の数値レポート（売上・CVR・CPA）
- Q2予算案（3パターン）
```

「前回からの持ち越し」は、 `areas/meeting-notes/` に保存された過去の議事録からアクションアイテムを自動抽出しています。会議を繰り返すほど、この持ち越し管理の精度が上がっていきます。

#### 会議後: 議事録

会議終了後、メモや書き起こしテキストを渡すと議事録に変換します。

```text
あなた: この会議の議事録を作って

A社ミーティング
- Q1は目標の92%達成
- Q2予算はプランBの500万で承認
- 新LP初稿は4/10まで、レビューは4/20
- 佐々木さんが社内稟議を4/3までに通す
- 次回は4/14 15時
```

出力では「決定事項」「アクションアイテム」「次回予定」が構造化されます。特にアクションアイテムは担当者・期限つきのテーブルになるため、見落としが防げます。

```markdown
### アクションアイテム
| 担当 | タスク | 期限 |
|------|--------|------|
| 田中 | LP初稿を作成 | 4/10 |
| 佐々木様 | 社内稟議を通す | 4/3 |
```

アクションアイテムの中に自分のタスクがあれば、AI秘書がプロジェクトへの登録を提案してくれます。会議で決まったことが自然にタスク管理に流れる仕組みです。

### 機能4: リサーチ

Webやドキュメントを調査し、ソース付きの構造化レポートを作成します。

```text
あなた: リモートワークツールを比較調査して
```

AI秘書はまず「何を・なぜ・どこまで調べるか」を1文で宣言してから調査を開始します。これは意図のズレを防ぐためのプロトコル設計です。

出力では、比較対象がテーブルで整理され、最後にアクション案が提示されます。

```markdown
## リサーチ: リモートワークツール比較（2026-03-30）

### 調査スコープ
チーム（10〜30名規模）向けのリモートワークツールを
ビデオ会議・チャット・タスク管理の3軸で比較する。

### 要約
コスパ重視ならGoogle Workspace、
機能統合重視ならMicrosoft 365が最適。

### 詳細

#### ビデオ会議
| ツール | 月額/人 | 最大参加者 | 録画 |
|--------|--------|----------|------|
| Google Meet | ¥680〜 | 100人 | 有 |
| Zoom | ¥1,370〜 | 300人 | 有 |
| Teams | ¥750〜 | 300人 | 有 |

### アクション案
1. Google Workspace を導入（理由: 既にGmail利用中、追加コスト最小）
2. Slack + Zoom の組み合わせ（理由: チャット品質重視の場合）

### ソース
- Google Workspace 料金ページ
- Zoom 料金ページ
```

品質ルールとして、事実と推測を分けること、ソースを必ず記載することがprotocols/research.mdに定義されています。「なんとなくそう思います」ではなく、根拠ベースのレポートが出てくる仕組みです。

### 機能5: プロジェクト管理

ここが他のAI秘書テンプレートとの差別化ポイントです。

仕事を依頼すると、AI秘書が自動的にプロジェクトとして登録し、成果物・参考資料・メモをフォルダで管理します。

#### プロジェクトの作成

```text
あなた: 来月のセミナー用のプレゼン資料を作って
```

この一言で、AI秘書は裏側で以下を自動実行します。

<iframe src="https://qiita.com/embed-contents/mermaid#qiita-embed-content__2d653ac63568fd19694681f088575ef6" frameborder="0"></iframe>

```text
projects/PRJ-001_セミナープレゼン/
├── README.md       ← 依頼内容・ステータス・進捗
├── output/         ← 成果物がここに入る
├── resources/      ← 参考資料がここに入る
└── notes/          ← 作業メモがここに入る
```

#### 一覧の確認

```text
あなた: 今のプロジェクト一覧を見せて
```

```markdown
| 番号 | タイトル | ステータス | 納期 |
|------|---------|----------|------|
| PRJ-001 | セミナープレゼン | 進行中 | 4/15 |
| PRJ-002 | Webサイトリニューアル | 進行中 | 5/1 |
| PRJ-003 | 競合調査レポート | 未着手 | 4/10 |
```

#### 追加作業

同じ案件への追加依頼は、AI秘書が既存プロジェクトに紐づけます。

```text
あなた: セミナープレゼンに、Q1の実績データも追加して
```

新しいフォルダは作られず、 `PRJ-001_セミナープレゼン/` に追加されます。

#### バージョン管理

修正版が出た場合はバージョン番号で管理されます。

```text
projects/PRJ-001_セミナープレゼン/
└── output/
    ├── v1_セミナープレゼン.md    ← 初稿
    └── v2_セミナープレゼン.md    ← 修正版
```

#### 完了とアーカイブ

```text
あなた: セミナープレゼンの件は完了で
```

AI秘書がステータスを「完了」に更新し、フォルダを `archive/` に移動します。 `projects/` には進行中の案件だけが残り、完了済みは `archive/` でいつでも振り返れます。

<iframe src="https://qiita.com/embed-contents/mermaid#qiita-embed-content__146d0a5d9feee3061a514bf03686c617" frameborder="0"></iframe>

この「依頼 → 自動登録 → 作業 → 完了 → アーカイブ」のライフサイクルが、PARAメソッドのProjects → Archiveの流れと一致しています。ファイル管理のルールを意識しなくても、自然と整理された状態が維持される仕組みです。

### 機能6: AI学習・記憶

この学習機能は、Claude Code公式の自動メモリ機能（ `~/.claude/` 配下の `MEMORY.md` ）とは別のものです。このテンプレート独自の仕組みとして、 `memory/` フォルダにマークダウンファイルで学習記録を蓄積します。

このテンプレートでは、学習メモを蓄積して次回の対応に反映しやすくする仕組みを用意しています。特別な操作は不要で、2つの仕組みが動きます。

#### セッション終了時の自動記録

「ありがとう」「終わり」などでセッションを閉じる際、AI秘書がその日の対話から学んだことを `memory/learnings.md` に追記しようとします。これはCLAUDE.mdの指示に基づく動作であり、Claude Codeのhook機能のような保証された自動処理ではありません。確実に記録したい場合は、「今日の学びを記録して」と明示的に依頼するのが確実です。

```markdown
### 2026-03-30

- **修正**: メールの返信案がフォーマルすぎた。カジュアルな文体を好む
- **発見**: カレンダーAPIの日時指定でタイムゾーンを明示する必要がある
- **改善**: リサーチレポートは表を多用した方が読みやすいとのフィードバック
```

#### 積極的学習（常時）

セッション中も、ユーザーの行動パターンを自動検出して `memory/patterns.md` に記録します。

- 繰り返し使う指示パターン（毎朝ブリーフィングを依頼している等）
- 修正フィードバック（「もっと短く」「表にして」等）
- 好みの傾向（箇条書きを好む、数字にはグラフを求める等）

次回のセッション開始時にmemory/を読み込むことで、過去の学習が反映されやすくなります。たとえば、初回は詳細すぎるレポートを出して「もっと短く」と言われた場合、そのフィードバックがmemory/に記録されていれば、2回目以降は簡潔なレポートを出しやすくなります。

## カスタマイズ方法

このテンプレートは3箇所でカスタマイズできます。

### 1\. config/ — 自分の設定を変える

`config/profile.md` をテキストエディタで開いて、名前・役職・仕事スタイルを編集します。AI秘書に「プロフィールを更新して」と話しかけても変更できます。

`config/tools.md` で使用ツールの有効/無効を切り替えます。

```markdown
変更前: - **状態**: 無効
変更後: - **状態**: 有効
```

### 2\. protocols/ — AI秘書の行動を変える

protocols/フォルダの各ファイルがAI秘書の「業務マニュアル」です。テキストエディタで日本語を書き換えるだけで、AIの行動が変わります。

たとえば、メールの優先度基準を変えたければ `protocols/email-triage.md` を編集します。「特定の取引先からのメールは常に高優先にする」「ニュースレターは無視に分類する」といったカスタマイズが可能です。

### 3\. memory/ — 学習をリセットする

`memory/learnings.md` と `memory/patterns.md` の中身を削除すれば、AI秘書がゼロから学び直します。

`CLAUDE.md` はAI秘書の動作の根幹です。大幅に変更すると正しく動作しなくなる場合があります。カスタマイズは `config/` と `protocols/` で行うことを推奨します。

## 設計で意識したこと

最後に、このテンプレートの設計で意識したことをまとめます。

### CLAUDE.mdは「ルーター」に徹する

CLAUDE.mdに全ての指示を書くと、ファイルが肥大化してAIの精度が下がります。CLAUDE.mdは65行に抑え、「行動原則」と「どのプロトコルを使うかの判断表」だけを持たせました。具体的な手順はprotocols/に分離することで、各機能を独立して改善できるようにしています。

### ツール連携がなくても動く

MCP連携が前提だと、セットアップの敷居が高くなります。すべての機能に「ツールが無効の場合」のフォールバックを用意し、コピペで情報を渡すだけでも動くようにしました。

### 個人情報はテンプレートに含めない

config/profile.mdは全項目が「（未記入）」の状態で配布しています。cloneした時点では誰の情報も入っていません。.gitignoreでmemory/やareas/、生成されたプロジェクトフォルダを除外し、運用中の個人データがGitHubにプッシュされることを防いでいます。なお、config/profile.mdとconfig/tools.mdはテンプレートとして追跡対象にしていますが、オンボーディング後に個人情報が入るため、必要に応じて.gitignoreに追加することを推奨します。

AI秘書テンプレートの全体像をまとめます。
